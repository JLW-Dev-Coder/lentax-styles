## Round-25/27/28 override rules currently active in vlp-default.css

Pasted verbatim from `themes/vlp-default.css` (HEAD 718ea30). Two section headers, three rule
blocks: (1) the portal page-bg rule + the round-27 content-block-wrapper clear, and (2) the
heading/text contrast rule. The `--vlpd-text-on-dark` token referenced below resolves to `#ffffff`
(round 27/28); the `#f5f5f5` literals are inert fallbacks.

```css
/* ============================================================
   PORTAL PAGE BACKGROUND — SD page-settings override
   Added 2026-06-21 — palette purge to monochrome sprint (§2.5)
   ============================================================
   SuiteDash "Page Styling Options → Primary Background Color"
   injects an inline <style> rule on every portal page with
   !important targeting body / .main-wrapper / .wrapper.
   That injection lands after our <link> tags in parse order
   and wins same-specificity !important ties.

   We lift specificity by chaining .page-with-styling-options
   onto body, beating SD's (0,1,1) selector at (0,2,1) and
   SD's (0,2,1) wrapper selectors at (0,3,1).

   This makes vlp-default.css the source of truth for portal
   page-bg color, removing per-view drift caused by individual
   SD admin settings. No background-image layer is set here:
   --lentax-page-bg-image resolves empty and the SD injection
   sets no image either, so the image layer is left untouched.

   Round 25 extension (2026-06-21): #client-page-view is a
   deeper portal wrapper (DOM position [9], between .main-wrapper
   and the visible content cards) painted white by platform CSS
   — it has NO inline style, so the round-24 .main-wrapper JS
   transparent override does not reach it. Added to this same
   override block to keep portal page-bg consolidated in one
   rule; the body.page-with-styling-options chain + the
   #client-page-view ID give (1,1,0,1) with !important, beating
   any platform rule.

   Round 27 extension (2026-06-21): broad clear of SD content-block
   wrappers (.cbe-content, .content-block-wrapper,
   .content-block-view-wrapper, .bg-widget, .widget-holder,
   .columns-form-wrap, .content-wrapper) to transparent — covers
   views 169143, 169144, 168981, 169141, 169146 which use different
   SD layout templates than 168967. Round 25 only targeted
   #client-page-view; these other templates paint a different
   wrapper white. Transparent (not #1a1a1a) so body remains the
   single source of truth for page-bg color.
   ============================================================ */

body.page-with-styling-options:not(.sd-dark-mode),
body.page-with-styling-options:not(.sd-dark-mode) .main-wrapper,
body.page-with-styling-options:not(.sd-dark-mode) .wrapper,
body.page-with-styling-options:not(.sd-dark-mode) #client-page-view {
    background-color: var(--vlpd-page-bg, #1a1a1a) !important;
}

/* Round 27 extension: clear ALL SD content-block wrappers to
   transparent so the body #1a1a1a paints through. Round 25 only
   targeted #client-page-view, but other portal views use
   different SD layout templates whose wrappers (.cbe-content,
   .content-block-wrapper, .bg-widget, .widget-holder,
   .columns-form-wrap) also paint white from platform CSS.

   Affected views: 169143, 169144, 168981, 169141, 169146.

   Transparent (not #1a1a1a) so the body remains the single
   source of truth for page-bg color. */
body.page-with-styling-options:not(.sd-dark-mode) .cbe-content,
body.page-with-styling-options:not(.sd-dark-mode) .content-block-wrapper,
body.page-with-styling-options:not(.sd-dark-mode) .content-block-view-wrapper,
body.page-with-styling-options:not(.sd-dark-mode) .bg-widget,
body.page-with-styling-options:not(.sd-dark-mode) .widget-holder,
body.page-with-styling-options:not(.sd-dark-mode) .columns-form-wrap,
body.page-with-styling-options:not(.sd-dark-mode) .content-wrapper {
    background-color: transparent !important;
    background-image: none !important;
}
```

```css
/* ============================================================
   PORTAL + DASHBOARD HEADING/TEXT CONTRAST OVERRIDE
   ============================================================
   Round 23 palette purge collapsed page bg to #1a1a1a but left
   the .styling-options-content wrapper's inherited text color at
   rgb(26,26,26) — same value. Headings + paragraph text became
   invisible (dark-on-dark, zero contrast). Affected surfaces:
   /dashboard (#dashboard-view) and all portal views
   (#client-page-view).

   This block flips inherited solid text to #f5f5f5 on:
     - Headings (h1-h6) inside .styling-options-content
     - .text-animated-waterfall (and its child spans)
     - Plain <span> and <p> elements whose color is purely
       inherited (no background-clip: text override)

   Excluded by design:
     - Elements with background-clip: text (orange gradient spans
       work because their fill-color is transparent, this rule
       does not touch them)
     - .dropdown-card, .dropdown-card-dark (dropdown menus carry
       their own dark/light theming)
     - Card interiors that already set their own color via inline
       style (color: #f5f5f5 from lentax-vlp-default.js)
     - Announcement banners and any element with explicit bg

   Specificity: body.page-with-styling-options chained with the
   .styling-options-content + child selector chain gives (0,3,1)
   per rule, beating the (0,1,0) inherit on the wrapper.
   ============================================================ */

body.page-with-styling-options .styling-options-content h1,
body.page-with-styling-options .styling-options-content h2,
body.page-with-styling-options .styling-options-content h3,
body.page-with-styling-options .styling-options-content h4,
body.page-with-styling-options .styling-options-content h5,
body.page-with-styling-options .styling-options-content h6,
body.page-with-styling-options .styling-options-content .text-animated-waterfall,
body.page-with-styling-options .styling-options-content .text-animated-waterfall span,
body.page-with-styling-options .styling-options-content .wysiwyg-content p,
body.page-with-styling-options .styling-options-content .wysiwyg-content > span {
    color: var(--vlpd-text-on-dark, #f5f5f5) !important;
    -webkit-text-fill-color: var(--vlpd-text-on-dark, #f5f5f5) !important;
}

/* Preserve gradient-text elements: any descendant span whose
   background-clip is set to text continues to display its own
   gradient/color via -webkit-text-fill-color: transparent. The
   rule above sets fill-color on h1/h2/.text-animated-waterfall,
   but the orange children override fill-color at higher
   specificity via their own inline-style or class chain. */
```
